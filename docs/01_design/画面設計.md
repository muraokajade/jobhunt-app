# 1. 概要

JobHunt Liteの画面設計では、React / TypeScript側の画面構成、状態管理、コンポーネント責務、API通信、認証画面、企業管理画面、詳細モーダルの仕様を整理する。

JobHunt Liteは、複数ページに分けるのではなく、1画面完結型のSPAとして構成する。

ログイン前は、ログイン / ユーザー登録画面を表示する。

ログイン後は、企業管理画面を表示し、以下の操作を1画面で行えるようにする。

- 企業登録
- 企業一覧表示
- 検索・絞り込み
- SummaryCards表示
- ActionLists表示
- 詳細モーダル編集
- priority / status インライン更新
- いいね切り替え
- 削除
- ログアウト

---

# 2. 目的

画面設計の目的は、転職活動中に必要な情報を1画面で素早く確認・更新できるようにすることである。

転職活動では、以下の情報を頻繁に確認する。

- 応募した企業
- 現在の選考状況
- 志望度
- 面談日
- 次アクション
- 書類・面接結果
- 求人URL
- 面談URL
- 落選状態
- 注目企業

そのため、JobHunt Liteでは、一覧・検索・集計・詳細編集を同じ画面上にまとめ、Notionよりも速く、Excelよりも状態管理しやすいUIを目指す。

# 3. 画面全体構成

## 3.1 ログイン前

authTokenが存在しない場合、ログイン / ユーザー登録画面を表示する。

表示要素は以下。

- JobHunt Liteタイトル
- 説明文
- 名前入力欄
- メールアドレス入力欄
- パスワード入力欄
- ログイン / 登録ボタン
- ログイン画面 / 登録画面の切り替えボタン
- Toast通知

名前入力欄は、authModeがregisterのときのみ表示する。

## 3.2 ログイン後

authTokenが存在する場合、企業管理画面を表示する。

表示順は以下。

1. AppHeader
2. Hero / 操作エリア
3. SummaryCards
4. SearchForm
5. ActionListsアコーディオン
6. CompanyRegisterForm または CompanyTable
7. CompanyDetailModal
8. Toast通知

# 4. 認証画面設計

## 4.1 authTokenによる表示切り替え

App.tsxでは、authTokenの有無で表示画面を切り替える。

    if (!authToken) {
      return ログイン / ユーザー登録画面;
    }

    return 企業管理画面;

## 4.2 authMode

authModeはログイン画面とユーザー登録画面を切り替えるための状態である。

    const [authMode, setAuthMode] = useState<"login" | "register">("login");

## 4.3 authForm

authFormは認証フォームの入力値を管理する。

    const [authForm, setAuthForm] = useState({
      name: "",
      email: "",
      password: "",
    });

## 4.4 login

login関数は、POST /api/login を呼び出す。

ログイン成功時は、以下を行う。

- tokenをlocalStorageに保存する
- authTokenにtokenをセットする
- authUserにユーザー情報をセットする
- Toastでログイン成功を表示する

## 4.5 register

register関数は、POST /api/register を呼び出す。

登録成功時は、以下を行う。

- tokenをlocalStorageに保存する
- authTokenにtokenをセットする
- authUserにユーザー情報をセットする
- Toastでユーザー登録成功を表示する

## 4.6 logout

logout関数は、POST /api/logout を呼び出す。

ログアウト時は、以下を行う。

- Laravel側でTokenを削除する
- localStorageからjobhunt_tokenを削除する
- authTokenをnullにする
- authUserをnullにする
- companiesを空にする
- dashboardをnullにする
- Toastでログアウト成功を表示する

# 7. useEffect設計

App.tsxでは、authTokenが変化したときに以下を実行する。

- fetchMe
- fetchCompanies
- fetchDashboard

authTokenがない場合は、companiesとdashboardをクリアする。

    useEffect(() => {
      if (!authToken) {
        setCompanies([]);
        setDashboard(null);
        return;
      }

      fetchMe(authToken);
      fetchCompanies(authToken);
      fetchDashboard(authToken);
    }, [authToken]);

## 7.1 設計意図

ログイン成功後、authTokenがセットされると自動でユーザー情報・企業一覧・Dashboard情報を取得する。

ログアウト時はauthTokenがnullになるため、画面上の企業データとDashboardデータを消す。

# 8. API通信設計

## 8.1 API_BASE_URL

APIのベースURLは以下。

    const API_BASE_URL = "http://127.0.0.1:8000/api";

## 8.2 createHeaders

createHeadersはAPI通信時のheadersを作る関数である。

tokenがある場合はAuthorizationヘッダーを付ける。

bodyがある場合はContent-Typeを付ける。

    function createHeaders(token: string | null, hasBody = false) {
      return {
        ...(hasBody ? { "Content-Type": "application/json" } : {}),
        Accept: "application/json",
        ...(token ? { Authorization: `Bearer ${token}` } : {}),
      };
    }

## 8.3 設計意図

認証必須APIではBearer Tokenが必要である。

createHeadersに集約することで、各API通信でheadersの書き忘れを防ぐ。

# 10. Hero / 操作エリア設計

## 10.1 表示内容

ログイン後のメイン画面上部には、Hero / 操作エリアを表示する。

表示内容は以下。

- JobHunt Liteラベル
- 応募企業を管理するタイトル
- 説明文
- ログイン中ユーザー名
- 登録ボタン
- 一覧ボタン
- ログアウトボタン

## 10.2 登録 / 一覧切り替え

mainPanelで表示内容を切り替える。

    type MainPanel = "register" | "list";

    const [mainPanel, setMainPanel] = useState<MainPanel>("list");

表示仕様は以下。

- mainPanelがregisterの場合、CompanyRegisterFormを表示する
- mainPanelがlistの場合、CompanyTableを表示する
- 初期表示はlist

## 10.3 設計意図

企業登録フォームと企業一覧を同時に大きく表示すると、画面密度が高くなりすぎる。

そのため、登録 / 一覧を切り替える。

登録後は一覧表示へ戻し、追加された企業をすぐ確認できるようにする。

# 11. AppHeader設計

## 11.1 役割

AppHeaderは、アプリ上部のヘッダーを表示するコンポーネントである。

主な表示要素は以下。

- JobHuntロゴ
- 転職活動CRMラベル
- ナビゲーション
- ユーザー名
- ログアウトボタン

## 11.2 現在の注意点

現在貼られているAppHeaderは、仮ユーザー情報を内部に持つ見た目用コンポーネントになっている。

一方で、App.tsx側ではauthUserとlogout関数を管理している。

そのため、現状ではログアウト処理はHero / 操作エリア側のログアウトボタンで実行している。

## 11.3 今後の改善候補

AppHeaderをより実装状態に合わせるなら、以下のpropsを受け取る形にする。

- authUser
- onLogout

これにより、AppHeader内の仮ユーザー情報をなくし、ログイン中ユーザー名とログアウト処理をApp.tsxから渡せる。

# 12. SummaryCards設計

## 12.1 役割

SummaryCardsは、転職活動状況の件数をカード形式で表示するコンポーネントである。

## 12.2 表示項目

- 応募総数
- 面談予定
- 返答待ち
- 内定
- 落選

## 12.3 props

SummaryCardsは以下を受け取る。

- companies
- dashboardSummary

## 12.4 データ取得方針

Dashboard APIからsummaryを取得できている場合は、dashboardSummaryを優先して表示する。

dashboardSummaryがない場合は、companiesからReact側で件数を計算する。

## 12.5 fallback設計

fallbackでは、companies配列からstatusごとの件数を数える。

    countByStatus(companies, "面談予定")
    countByStatus(companies, "面談後返答待ち")
    countByStatus(companies, "内定")
    countByStatus(companies, "落選")

## 12.6 注意点

Dashboard設計ではsummary.highPriorityも存在する。

現在のSummaryCards表示では高優先度カードは表示していない。

必要であれば、SummaryCardsに高優先度カードを追加する。

# 14. ActionLists設計

## 14.1 役割

ActionListsは、次に確認すべき企業を表示するコンポーネントである。

表示リストは以下。

- 面談予定
- 確認待ち
- 高優先度

## 14.2 props

ActionListsは以下を受け取る。

- companies
- dashboardActionLists
- onOpenDetail

## 14.3 表示方式

App.tsx側では、ActionListsをアコーディオン形式で表示する。

isActionListsOpenがtrueの場合のみ、ActionLists本体を表示する。

## 14.4 データ取得方針

Dashboard APIからactionListsを取得できている場合は、dashboardActionListsを優先する。

dashboardActionListsがない場合は、companiesからReact側でfallback計算する。

## 14.5 fallback関数

ActionListsには以下のfallback関数がある。

- getInterviewCompanies
- getWaitingCompanies
- getHighPriorityCompanies
- isClosedCompany

## 14.6 詳細モーダル連携

ActionLists内の詳細ボタンを押すと、onOpenDetail(company)を呼ぶ。

App.tsxでは、onOpenDetailとしてopenDetailModalを渡している。

これにより、ActionListsからでもCompanyDetailModalを開ける。

## 14.7 重要注意点

現在のgetInterviewCompaniesには注意が必要である。

    function getInterviewCompanies(companies: Company[]): Company[] {
        return companies.filter((company) => {
            return company.status === "面談予定" || company.status !== null
        });
    }

この条件だと、statusがnullでない企業のほとんどが面談予定に含まれる可能性がある。

本来の意図は、以下のどちらかに近い。

    company.status === "面談予定"

または、

    company.status === "面談予定" || company.interviewDate !== null

ただし、Dashboard APIのactionListsを優先して使っているため、通常運用ではLaravel側の結果を優先する。

fallbackとして残す場合は、条件修正を検討する。

# 19. detailForm初期化設計

## 19.1 openDetailModal

openDetailModalでは、選択したCompanyをselectedCompanyにセットし、CompanyResource形式のcamelCaseデータをCompanyForm形式へ詰め替える。

CompanyはReact側ではcamelCaseで扱う。

CompanyFormはAPI送信用にsnake_caseを含む。

そのため、openDetailModalで以下のように変換する。

- company.appliedDate → detailForm.applied_date
- company.interviewDate → detailForm.interview_date
- company.jobUrl → detailForm.job_url
- company.interviewUrl → detailForm.interview_url
- company.nextAction → detailForm.next_action
- company.documentResult → detailForm.document_result
- company.firstInterviewResult → detailForm.first_interview_result
- company.secondInterviewResult → detailForm.second_interview_result
- company.finalResult → detailForm.final_result
- company.rejectionStage → detailForm.rejection_stage

## 19.2 日時変換

interviewDateは、toDateTimeLocalでdatetime-local入力用に変換する。

保存時は、toApiDateTimeでAPI送信用に変換する。

## 19.3 closeDetailModal

closeDetailModalでは以下を行う。

- isDetailOpenをfalseにする
- selectedCompanyをnullにする
- detailFormを初期状態へ戻す

# 25. 現在の注意点・改善候補

## 25.1 AppHeaderの認証連携

AppHeaderは仮ユーザー情報を内部で持っているため、authUserとlogoutをpropsで受け取る形に改善できる。

## 25.2 SummaryCardsの高優先度表示

Dashboard APIにはsummary.highPriorityがあるが、現在のSummaryCardsでは高優先度カードを表示していない。

必要であれば、6枚目のカードとして追加する。

## 25.3 ActionLists fallback条件

getInterviewCompaniesのfallback条件が広すぎる。

現在の条件。

    company.status === "面談予定" || company.status !== null

この条件だと、statusがnullでない企業が多く含まれる可能性がある。

修正候補。

    company.status === "面談予定" || company.interviewDate !== null

## 25.4 API_BASE_URLの環境変数化

現在はAPI_BASE_URLが固定である。

    http://127.0.0.1:8000/api

将来的には環境変数化を検討する。

## 25.5 エラー詳細表示

現在は失敗時にToastで簡易メッセージを表示している。

将来的にはLaravelのバリデーションエラー内容をフォーム下に表示することを検討する。

# 26. 一言まとめ

JobHunt Liteの画面設計では、App.tsxを親として、認証状態、企業一覧、検索条件、登録フォーム、詳細モーダル、Dashboard、Toastを管理する。

子コンポーネントは表示と入力イベント通知に集中し、API通信はApp.tsxに集約する。

ログイン前は認証画面を表示し、ログイン後は企業管理画面を表示する。

企業管理画面では、SummaryCards、SearchForm、ActionLists、CompanyRegisterForm、CompanyTable、CompanyDetailModalを組み合わせ、転職活動の応募企業管理を1画面で完結させる。
